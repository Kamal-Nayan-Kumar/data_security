 **Secure Package Repository System**

**Course:** Data Security & Privacy  
**Project Type:** Full-Stack Application with Security Focus

---

**Summary**

This project involves building a secure, APT-like package repository system that enables developers to distribute software packages and users to download them safely. The system implements multiple layers of cybersecurity controls including GPG-based package signing, integrity verification, malware scanning, and secure authentication mechanisms. The platform consists of dual web dashboards (user and developer), a command-line interface tool, and an optional CI/CD pipeline for automated package publishing.

---

**Problem Statement**

**Current Challenges in Software Distribution**

Software distribution faces several critical security challenges that affect both developers and end-users\[1\]\[2\]:

1. **Package Tampering** — Attackers can modify software packages during transit or on compromised servers, injecting malicious code without user knowledge

2. **Lack of Authenticity Verification** — Users have difficulty verifying whether downloaded software actually comes from the claimed developer

3. **Supply Chain Attacks** — Compromised developer accounts or build systems can distribute malware to thousands of users automatically\[3\]

4. **Centralized Trust Issues** — Many small-scale repositories lack proper security infrastructure, making them vulnerable targets

5. **Manual Verification Burden** — Developers and users must manually verify checksums and signatures, which is error-prone and often skipped

**Project Goals**

This project aims to:

1. Build a production-ready package repository with enterprise-grade security features

2. Implement asymmetric cryptography for package authentication and integrity

3. Create automated security scanning to detect malicious code patterns

4. Provide both graphical and command-line interfaces for different user workflows

5. Document security architecture decisions and threat mitigation strategies

**Proposed Solution**

**System Overview**

The solution is a three-tier architecture consisting of:

1. **Frontend Layer** — Two React-based dashboards (user browsing interface and developer management portal)

2. **Backend Layer** — Python FastAPI server handling authentication, package management, signature verification, and API endpoints

3. **CLI Tool Layer** — Python command-line application for installing packages and pushing releases

**Core Functionality**

**User Workflow:**

1. User registers and receives authentication credentials

2. Browses available packages via web dashboard or searches using keywords

3. Views package details including description, version history, and security status

4. Downloads packages using CLI tool: myrepo install package-name

5. CLI automatically verifies GPG signature and checksum before installation

**Developer Workflow:**

1. Developer registers and generates GPG keypair

2. Uploads public key to repository server

3. Pushes package using CLI: myrepo push package-name-v1.0.tar.gz

4. CLI signs package with private key and uploads to server

5. Server runs automated security scans and publishes if checks pass

6. Optional: Connect GitHub repository for CI/CD automated publishing on release tags

**Cybersecurity Techniques Used**

**1\. Asymmetric Cryptography (GPG Signing)**

**Concept:** Uses public-private key pairs where the private key signs packages and the public key verifies signatures\[5\].

**Implementation:**

1. Each developer generates a GPG keypair using python-gnupg library

2. Private key remains on developer's machine (never transmitted)

3. Public key uploaded to repository server during registration

4. Before upload, CLI creates detached signature: signature \= gpg.sign\_file(package, keyid=private\_key)

5. Server stores package \+ signature \+ public key

6. On download, CLI verifies: gpg.verify\_file(signature, public\_key)

7. If signature invalid → installation aborted

**Security Benefit:** Even if attacker compromises server and modifies packages, they cannot forge valid signatures without the developer's private key\[6\].

**2\. Cryptographic Hash Functions (SHA-256)**

**Concept:** One-way mathematical function that produces unique fixed-size fingerprint of file content\[7\].

**Implementation:**

1. On upload: checksum \= hashlib.sha256(file\_bytes).hexdigest()

2. Stored in database alongside package metadata

3. On download: Client recomputes hash and compares

4. Single bit change produces completely different hash (avalanche effect)

**Security Benefit:** Detects accidental corruption and intentional tampering. Provides integrity verification independent of GPG signatures\[8\].

**3\. Transport Layer Security (HTTPS/TLS)**

**Concept:** Encrypts all network communication between client and server\[9\].

**Implementation:**

1. Backend configured with SSL certificate (Let's Encrypt for production)

2. All API endpoints accessible only via HTTPS

3. CLI tool enforces HTTPS and rejects insecure connections

4. Implements certificate pinning for critical operations

**Security Benefit:** Prevents Man-in-the-Middle (MITM) attacks where attacker intercepts network traffic to swap packages during download\[10\].

**4\. JSON Web Tokens (JWT) Authentication**

**Concept:** Stateless authentication using cryptographically signed tokens\[11\].

**Implementation:**

1. User login generates JWT containing user ID, role, expiration time

2. Token signed with server secret key using HMAC-SHA256

3. Client includes token in Authorization header for API requests

4. Server validates signature and checks expiration before processing request

5. Tokens expire after 24 hours, requiring re-authentication

**Security Benefit:** Prevents unauthorized package uploads and account hijacking. Stateless design scales well and limits damage from stolen tokens\[12\].

**5\. Static Malware Analysis**

**Concept:** Automated scanning of code without execution to detect malicious patterns\[13\].

**Implementation:**

1. For Python packages: Bandit scanner checks for dangerous functions (eval(), exec(), unsafe pickle usage)

2. Pattern matching for suspicious behaviors: reverse shells, obfuscated code, network callbacks to unknown IPs

3. Entropy analysis to detect encrypted/packed malware

4. YARA rules for known malware signatures

5. Packages flagged as suspicious require manual review before publishing

**Security Benefit:** Catches common malicious code patterns before they reach users, similar to PyPI's automated security scanning\[14\].

**6\. Signed Release Manifests**

**Concept:** Cryptographically signed index of all packages in repository\[15\].

**Implementation:**

1. Server maintains Release file containing list of all packages with checksums

2. File signed with repository master key: Release.gpg

3. CLI downloads manifest first, verifies signature, then checks package against manifest

4. Prevents rollback attacks where attacker serves old vulnerable versions

**Security Benefit:** Protects repository-level integrity. Attacker cannot selectively hide security updates or serve unauthorized packages\[16\].

**7\. Rate Limiting and DoS Protection**

**Concept:** Restrict number of requests from single IP to prevent abuse\[17\].

**Implementation:**

1. FastAPI middleware with Redis-backed rate limiter

2. Login endpoint: 5 attempts per 15 minutes per IP

3. Download endpoint: 100 downloads per hour per IP

4. Upload endpoint: 10 uploads per day per developer account

5. Returns HTTP 429 (Too Many Requests) when limit exceeded

**Security Benefit:** Prevents credential stuffing attacks, bandwidth exhaustion, and automated malware distribution\[18\].

vget is a secure package repository and CLI system built with a focus on cryptographic verification and data integrity. It ensures that every package you download is exactly what the author intended to publish.

## Architecture

The system consists of two primary components:
- **Backend**: Built with Axum, using PostgreSQL (via SQLx) for persistence. It handles package metadata, user authentication (JWT), and storage.
- **CLI**: A command-line interface built with Clap that performs client-side cryptographic operations, including key generation, package signing, and verification.

## Security Features

vget implements several layers of security to ensure trust and integrity:
- **SHA256 Checksums**: Every package version is hashed during publication. The client re-calculates this hash upon download to detect any corruption or tampering.
- **Ed25519 Signatures**: Developers sign the package checksum using their private key. The client verifies this signature using the developer's public key before extraction.
- **JWT Authentication**: User sessions are protected using JSON Web Tokens.
- **Local Verification**: Cryptographic checks happen on your local machine, not just the server.

## Developer Guide

### Prerequisites
- Rust (latest stable)
- Docker and Docker Compose
- `sqlx-cli` (optional, for migrations)

### Setup
1. Start the database and storage infrastructure:
   ```bash
   docker compose up -d
   ```
2. Run migrations:
   ```bash
   cd backend && sqlx migrate run
   ```
3. Start the backend server:
   ```bash
   cargo run -p backend
   ```

### Build CLI Binary

Build the `vget` binary:

```bash
cargo build --release -p cli
```

The binary will be available at:

```bash
target/release/vget
```

### Publishing Packages
1. **Generate Keys**: Create your unique Ed25519 identity keys.
   ```bash
   target/release/vget keygen
   ```
2. **Register as a Developer**: Link your public key to a username.
   ```bash
   target/release/vget dev-register --username <your_name>
   ```
3. **Publish**: Sign and upload your package.
   ```bash
   target/release/vget publish --path <directory_or_file> --version <version>
   ```

## User Guide

### Authentication
Users can register and log in through the CLI:
```bash
# Register a user via CLI
target/release/vget register --username alice

# Log in via CLI
target/release/vget login --username alice
```

### Finding and Installing Packages
1. **Search**: Find packages by name or description.
   ```bash
   target/release/vget search <query>
   ```
2. **Install**: Download, verify, and extract a package.
   ```bash
   target/release/vget install <package_name>
   ```
   The CLI automatically verifies the SHA256 checksum and Ed25519 signature. If verification fails, the package will not be installed.
